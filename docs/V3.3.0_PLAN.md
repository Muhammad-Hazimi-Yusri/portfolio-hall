## Scope decision

| Option | Scope | Risk | Visitor value |
|---|---|---|---|
| (a) Captions only | Add captions to existing slideshow | Low | Medium — adds context to every painting |
| (b) Captions + iframe | Full v3.3.0 | High (see tradeoffs) | High but only on 2 sites |
| **(c) Phased** | **(a) now, iframe in v3.3.1** | **Low now** | **Banks the safe win, keeps stretch open** |

Going with **(c)**. Slideshow-with-captions is a clean, low-risk extension of
v3.0.0 — every painting gains more context immediately. Iframe is a genuine
1–2 week problem (matrix3d DOM overlay, CSP coordination, pointer-event
toggle, mobile/VR gating, occlusion edge cases) and only pays off on two
URLs; not worth blocking the v3.3.0 ship.

## Tradeoffs surfaced (informs deferral of iframe)

### Iframe-on-3D-plane is hard, in order of severity

1. **Cross-origin live iframes can't be sampled to a WebGL texture.** Browsers
   refuse `texImage2D(iframe)`. The only realistic path is a **DOM overlay**
   `<iframe>` positioned via `Vector3.Project` + CSS `transform: matrix3d(...)`
   over the canvas — which means the iframe is *not* in the 3D scene, just
   visually aligned with it.
2. **X-Frame-Options / CSP `frame-ancestors`**. Author owns both candidate
   sites, so `frame-ancestors 'self' https://muhammad-hazimi-yusri.github.io`
   is doable — but it's a per-site config change with no automated check.
3. **Pointer-event hijacking**. Overlay iframe steals all pointer input above
   it. Needs an explicit "Browse / Stop browsing" toggle; pointer-lock can't
   coexist with iframe focus. Free-roam controls would have to hand off.
4. **Camera movement during free-roam**. The CSS matrix has to update every
   frame; iframe content reflows under transform changes — expensive on
   integrated GPUs. Either freeze the iframe to "stationary camera only" or
   eat the perf cost.
5. **No occlusion**. DOM overlay always paints over the canvas — an avatar
   walking between camera and painting won't occlude it. Visually wrong.
6. **Mobile gate is correct**. Tiny touch screens + DOM-warp iframes = bad UX.
   `isMobile()` already exists in `src/utils/detection.ts:24`.
7. **VR**: WebXR doesn't render the page DOM, so iframes vanish in VR. The
   spec's "VR fallback to slideshow" lines up here — easy to implement once
   the iframe path exists.

These are deferrable, not unsolvable — but they justify shipping (a) first.

## v3.3.0 scope (this branch)

### Goal
Every painting becomes a self-narrating slideshow: each frame can carry a
short caption rendered onto a strip below the canvas. Captions are optional
per frame — paintings without captions look identical to v3.0.0.

### Data model change

`src/types/poi.ts` — extend `POIContent`:

```ts
// Backwards-compatible: paintings without captions just won't get a strip.
// Parallel arrays keep the JSON readable; index alignment is enforced at
// build / runtime with a warning if lengths mismatch.
thumbnails?: string[]
captions?: string[]   // NEW — optional, aligned by index with thumbnails
```

Alternative considered: `slides?: { src: string; caption?: string }[]`. Cleaner
type, but requires migrating every existing painting in `pois.json` even
though most won't need captions yet. Parallel arrays keep the diff minimal
and make caption-less paintings free.

### Component changes

1. **`src/3d/paintingSlideshow.ts`** — extend `SlideshowConfig`:
   - Add `captions?: string[]` and `captionMesh?: Mesh`.
   - When the slideshow swaps `currentIndex`, also update a `DynamicTexture`
     bound to `captionMesh.material.diffuseTexture` with the matching caption
     (re-using the same fade timing so caption fades in/out alongside the
     image — single state machine, no separate timer).
   - If `captions` is missing or all empty: skip caption mesh entirely.

2. **`src/3d/pois.ts`** — `createPaintingMesh`:
   - When `poi.content.captions?.some(Boolean)` is true, build a thin caption
     strip mesh (1.5 m wide × ~0.18 m tall, depth ~0.02 m) parented to the
     painting group, positioned just under the canvas frame. New material
     uses a `DynamicTexture` so we can rasterize text per-slide.
   - Return the strip on `slideshowTarget` so `BabylonScene.tsx` can pass it
     into `createSlideshow`.

3. **Caption rendering helper** (small new util, colocated in
   `paintingSlideshow.ts` or `pois.ts`):
   - `renderCaptionToTexture(tex: DynamicTexture, text: string)` — clear,
     fill semi-transparent background (matches museum dark theme), draw text
     with auto-wrap at ~36 chars / 2 lines max. Re-uses the same canvas-2D
     pattern as `applyFallbackTexture`.

4. **`src/data/pois.json`** — add `captions` arrays to the project paintings
   that have multi-frame thumbnails. Captions should be 4–7 words each (e.g.
   for `eee-roadmap`: ["Topic graph view", "Progress dashboard", "Cross-device
   sync"]). This is content work, not engineering.

5. **README update** — move v3.3.0 line in `Upcoming` into a new shipped-row
   under `Released` after merge (consistent with the v3.x changelog blocks).
   Note in the description that the iframe stretch is deferred.

### What's intentionally NOT in v3.3.0

- DOM-overlay iframe (deferred to v3.3.1 / unscheduled)
- Per-painting "Open live site" launcher beyond the existing `Live` link in
  the inspect modal (already there)
- Variable-length frame counts beyond what v3.0.0 already supports — N
  thumbnails just works
- Caption animations beyond the existing fade (no Ken Burns, no slide-in)

### Reuse map (don't duplicate)

| Need | Existing thing to reuse |
|---|---|
| Texture preloading + load-failure timeout | Lift the loop in `paintingSlideshow.ts:111–130` as-is |
| Crossfade animator | Same `Phase` state machine — extend to drive caption alpha in lockstep |
| Distance-based pause | Already in `createSlideshow`; new caption strip rides the same observer |
| Fallback dynamic texture | `applyFallbackTexture` pattern in `pois.ts:139` is the template for `renderCaptionToTexture` |
| Painting mesh + frame | `createPaintingMesh` in `pois.ts:45` |
| Wiring per-POI | `BabylonScene.tsx:280–289` already maps over `slideshowTargets` |

### Files touched

- `src/types/poi.ts` (+1 field)
- `src/3d/paintingSlideshow.ts` (extend config + caption fade)
- `src/3d/pois.ts` (caption strip mesh + render helper, return on

### Acceptance checks

- Paintings with `captions` show a captioned strip that fades with each frame
- Paintings without `captions` are visually identical to v3.2.5 (no strip,
  no extra mesh)
- Distance-pause still works (strip pauses with image, no drift)
- VR: caption strip is a regular mesh — renders correctly in WebXR by
  construction (no DOM, no iframe, no fallback needed at this stage)
- Mobile: nothing extra to gate — same Babylon mesh path
- `npm run lint` and `npm run typecheck` clean
- Manual: stand near each painting, confirm captions sync with frames; walk
  past the 30 m pause radius, confirm both image + caption freeze together

## v3.3.1 sketch (NOT in this PR — for reference)

If/when iframe ships:
- New POI flag `interactionConfig.iframeUrl?: string` (re-use the existing
  `interactionType: 'iframe'` slot).
- DOM overlay component: `<WebPanelOverlay>` rendered in `BabylonScene.tsx`,
  reads camera + plane mesh, computes screen-space corners with
  `Vector3.Project`, applies `matrix3d` per frame.
- Hard gates: `!isMobile() && !isInVR && cameraRef.current.isFlyingTo === false`.
- Explicit "Browse" toggle button → switches the painting plane to "iframe
  mode", releases pointer-lock, makes overlay pointer-events-auto. "Stop
  browsing" reverts.
- VR fallback: when `isInVR`, painting reverts to slideshow (already free —
  just don't mount the overlay in VR state).
- Author tasks: add `Content-Security-Policy: frame-ancestors` headers on
  `eee-roadmap.muhammadhazimiyusri.uk` and `food-wars.muhammadhazimiyusri.uk`
  before the feature can ship; never iframe `portfolio-hall` itself.
